# Dhandho_RSI30 구현 명세서

> **개정일**: 2026-06-30 · **상태**: Phase 2 완료(LLM 정성 그라운딩 가동) · **운영**: GitHub Actions 무서버 MVP
> 본 문서는 현재 코드베이스(`main` 기준)를 그대로 반영한 구현 명세서다. 설계 의도와 실제 동작이 일치하도록 유지한다.

---

## 0. 개요

한국 상장기업 중 **RSI 14일 ≤ 30** 으로 1차 필터링한 종목에, 단도투자(Mohnish Pabrai, *Dhandho*)
관점의 **정량·정성·정책(상법) 스코어링(A~F)** 을 적용해 **BUY / WATCH / PASS** 신호를 산출하고,
**LLM 정성 그라운딩까지 통과한 종목만** 텔레그램으로 알림한다.

- **자동매매 아님.** 최종 매수/매도 판단은 사람이 공식 근거로 수행하는 것을 전제로 설계.
- **무서버 MVP.** GitHub Actions 스케줄러 + SQLite(Actions 캐시 영속화)로 비용 0에 가깝게 운영.
- **핵심 원칙**: ① 임계값 하드코딩 금지(전부 `config.py`) ② 멱등(같은 날 재실행해도 중복 적재·알림 없음)
  ③ 근거 없는 정성 판단 금지(LLM은 DART 근거에만 기반, 근거 부족 시 자동 강등).

## 1. 파이프라인 (L0 ~ L8)

```
L0 데이터수집 → L1 RSI 1차필터 → L2~L6 단도 스코어링(A~F) → L7 게이트/신호 → L8 알림
```

| 레이어 | 내용 | 구현 |
|---|---|---|
| **L0** | KRX 전 종목 EOD OHLCV 일괄 수집 → SQLite(`prices`) + 유니버스 갱신 | `data/krx_eod.py`, `run_daily.collect_eod` |
| **L1** | Wilder 14일 RSI 산출 → `RSI≤30` + 유동성/품질 필터 → 후보 추출 | `indicators/rsi.py`, `run_daily.compute_rsi_and_filter` |
| **L2~L6** | DART 재무 기반 A~F 섹션 스코어링(정량) + 정성(정형 결정론 + 서술형 LLM 그라운딩) | `scoring/engine.py`, `scoring/sections.py`, `scoring/llm_grounding.py` |
| **L7** | 게이트(A≥3, D≥3, 플래그無) + 총점 → BUY/WATCH/PASS | `scoring/gate.py` |
| **L8** | 텔레그램 알림: BUY 우선, **BUY 0건이면 그라운딩 숏리스트 폴백** | `notify/telegram.py`, `run_daily._render_payload`/`_send_payload` |

## 2. L0 — 데이터 수집

- **소스**: KRX 공식 Open API (`https://data-dbg.krx.co.kr/svc/apis`), **POST(JSON 바디)** + `AUTH_KEY` 헤더.
  - 엔드포인트: 유가증권 `sto/stk_bydd_trd`, 코스닥 `sto/ksq_bydd_trd` (일별매매정보, `basDd=YYYYMMDD`).
  - 인증키 발급과 **별개로 데이터셋별 이용신청** 필요. 엔드포인트/필드는 `config.KRXConfig`에서 교체 가능.
- **적재**: `prices`(OHLCV·거래대금·시총·상장주식수) upsert. 가장 최근 거래일 스냅샷으로 `universe` 메타 갱신.
- **catch-up & 멱등**: `collection_log`로 이미 수집한 거래일은 스킵. 누락 거래일은 다음 실행에서 자동 보충.
- **유니버스 태깅**(`data/universe.py`): 금융/지주/리츠 분류, 순수 금융 등은 `excluded=1`로 스코어링 제외.

## 3. L1 — RSI 1차 필터 + 유동성/품질 필터

- **RSI**: Wilder 14일(`RSI_PERIOD=14`), 최소 데이터 `RSI_MIN_PERIODS=30` 미만이면 산출 제외. `prices.rsi14` 갱신.
- **1차 컷**: `RSI ≤ RSI_THRESHOLD(30)`.
- **유동성/품질 필터**(`config.LiquidityConfig`):
  - 보통주만(`LIQ_COMMON_ONLY=1`): 우선주·신주인수권·ELW 제외.
  - 최근 `LIQ_WINDOW(20)` 거래일 평균 거래대금 ≥ `LIQ_MIN_VALUE(1억 원)`.
  - 최신 거래일 데이터 누락(거래정지/상폐 등) 제외.
- 결과는 RSI 오름차순(가장 과매도 우선) 정렬.

## 4. L2~L6 — 단도 스코어링 (A~F)

각 섹션은 0~5점. **섹션 가중치**(`gate.SECTION_WEIGHTS`):

| 섹션 | 라벨 | 가중치 | 하위지표(가중치) |
|---|---|---|---|
| **A** | 질적우위(안전마진) | **0.25** | A1 순현금(.35) · A2 NCAV(.25) · A3 재무건전성(.25) · A4 FCF안정(.15) |
| **B** | 수익성 | **0.20** | B1 ROIC(.35) · B2 마진예측(.25) · B3 추세(.15) · **B4 정성해자(.25)** 🔶 |
| **C** | 저평가 | **0.20** | C1 트레일링멀티플(.35) · C2 과거밴드(.25)¹ · C3 이익추세(.25) · C4 과도낙폭(.15) |
| **D** | 안정성 | **0.15** | D1 매출이익추세(.30) · **D2 급락원인(.30)** 🔶 · **D3 산업사양화(.20)** 🔶 · D4 재무생존력(.20) |
| **E** | 주주환원 | **0.10** | E1 주주환원(.40) · E2 상법수혜(.35) · E3 촉매근접(.25) |
| **F** | 내부자 | **0.10** | **F1 자본배분(.35)** 🔶 · F2 내부자정렬(.30) · **F3 IR투명성(.35)** 🔶 |

🔶 = 서술형 **LLM 그라운딩** 항목(§7).  ¹ C2는 과거 시총 시계열 필요로 현재 미그라운딩(근거불충분 처리).

- **정량 지표**(`scoring/metrics.py`): 순현금·NCAV·ROIC·EV/EBIT·PER/PBR/PSR·FCF 안정성·마진 변동계수 등.
- **업종 상대평가**(`scoring/sector_relative.py`): ROIC·마진·밸류에이션은 동일 업종 내 백분위로 점수화(2-pass).
- **정성 — 정형 결정론**: `E1`(미소각 자사주비중+배당), `D4`(감사의견·계속기업), `F2`(내부자 순증감)는
  DART 정기보고서/소유보고에서 **결정론적**으로 산출. 부하 관리를 위해 가장 과매도된 상위 `QUAL_MAX(100)`개만 정성 적재.
- **정성 — 서술형 LLM 그라운딩**: B4·D2·D3·F1·F3 (§7).
- **근거불충분 규칙(§8)**: 정성 항목 근거가 없으면 점수 상한 **2.5 + 플래그**. 게이트 섹션(A·D) 플래그 시 BUY 불가.

## 5. L7 — 게이트 / 신호 (`scoring/gate.py`)

```
총점 = Σ (섹션가중치 × 섹션점수)

게이트 통과 ⟺ A≥3.0  AND  D≥3.0  AND  A·D에 근거불충분 플래그 없음

signal = BUY    if (게이트 통과 AND 총점 ≥ 4.0)
       = PASS   if (총점 < 3.0)
       = WATCH  otherwise
```

- **BUY ⟺ LLM 고확신 선정**: D 게이트는 D2/D3가 LLM으로 그라운딩되어야 통과 가능 → BUY는 사실상
  "정성 LLM 검증을 거친 고확신 후보". WATCH/PASS는 BUY 미달.

## 6. L8 — 알림 정책 (현행)

> 정책 핵심: **"LLM 정성 그라운딩을 거친 종목만 알림에 나간다. 단순 필터링 결과는 알림 금지."**

| 상황 | 발송 내용 |
|---|---|
| BUY ≥ 1건 | **BUY 알림만** (`format_buy`, 종목별 상세) |
| **BUY 0건** | **그라운딩 숏리스트 폴백** — `selection_reason`/`drop_reason`이 있는 **그라운딩 종목만** 다이제스트 발송 |
| `SEND_WATCH_DIGEST=1` | (점검용 옵트인) BUY 유무와 무관하게 전체 결과 다이제스트 동봉 |

- **폴백 다이제스트 형식**(3줄/종목): `종목명(코드) 총점·신호·RSI` / `하락사유` / `종목선정사유`.
  - **하락사유** = LLM D2(급락원인) 우선, 없으면 52주 고가 대비 낙폭.
  - **종목선정사유** = LLM 정성 문장(`selection_reason`). 점수 숫자 아님.
- **미그라운딩(단순 필터) 종목은 폴백에서도 제외** → 정책 요구를 코드로 강제.
- **중복 방지**: BUY는 `signals_log`(ticker,date,BUY), 다이제스트는 `__DIGEST__` 센티넬로 거래일 1회만.
- 렌더는 계산 단계가 수행하고 `notify_queue`에 페이로드(JSON `{buys, digest}`) 저장 → 전송 단계가 그대로 발송(§9).

## 7. LLM 정성 그라운딩 (§9 상세)

**대상 항목**: `B4`(해자) · `D2`(급락원인) · `D3`(산업) · `F1`(자본배분) · `F3`(IR). → 1~5 판정 + 근거 한 문장.

### 7.1 2모델 파이프라인 (추출 ↔ 스코어링)

```
DART 원문(사업의 내용·best-effort) + 최근 공시목록 + 정형 스냅샷
   │
   ├─[추출] Haiku 4.5 (CLAUDE_EXTRACT_MODEL)  → 항목별 '단서'만 압축 (비싼 토큰은 여기서만 소비)
   │
   └─[스코어링] Sonnet 4.6 (CLAUDE_MODEL)     → 정형 스냅샷 + 압축단서로 1~5 구조화 판정
                                              → score / grounded / reason + drop_reason + selection_reason
```

- 원문 발췌 실패 시 공시목록으로 폴백. 근거 부족 항목은 `grounded=false`로 반환 → 2.5상한+플래그 유지.
- 결과는 `qual_grounding` 테이블에 항목별 저장(사람 검토용). `ANTHROPIC_API_KEY` 없으면 자동 비활성.

### 7.2 토큰 절감 5축

1. **관련 근거만 입력** — 보고서 원문 전체가 아니라 구조화 스냅샷(재무·가격) + 공시 *제목*(절단).
2. **게이트 유망 후보만 호출** — A 통과(≥3·무플래그) + 정량총점 ≥ `LLM_MIN_TOTAL(3.5)`인 상위 `LLM_MAX(8)`개만.
3. **계산 시점 동기 자체완결** — 같은 런·같은 거래일에 그라운딩 완료(날짜 키 불일치 방지).
   Batch API(50%↓)는 두 런으로 쪼개면 날짜 키가 어긋나는 결함이 있어 **기본 비활성**(`LLM_BATCH=1`로 수동 옵션).
4. **프롬프트 캐싱** — 고정 채점 루브릭을 스코어링 system 블록에 캐시. 최소 프리픽스 Sonnet/Haiku=2048·Opus=4096 토큰.
5. **압축 출력** — 점수 + 짧은 사유(≤60자)만, `LLM_MAX_TOKENS(1500)`·`extract_max_tokens(700)` 하향(출력 토큰이 비쌈).

### 7.3 그라운딩 모드 (`engine.score_candidates`)

- `sync`(기본): 알림 런 안에서 동기 호출 → 재채점·저장.
- `db`: 사전 적용된 배치 결과를 DB에서 반영(+ 누락분 동기 보완 `LLM_SYNC_FALLBACK=1`).
- `off`: 비활성. 배치 제출은 `--submit-batch`(야간), 적용은 `apply_pending_batch`.

## 8. 무서버 운영 — 계산/전송 2단계 분리

> **문제**: GitHub cron은 UTC 고정(미국 DST 미반영) + best-effort라 수십 분~수 시간 지연(실측 8~11h).
> 또 "한국 장 마감일"과 "미국 장 마감 기준 전송일"의 **날짜 키 불일치**가 그라운딩 0건 버그를 유발했다.
> **해법**: 무거운 계산은 저녁에 끝내 큐에 저장하고, 가벼운 전송만 다음날 아침에 발송. 일자 하드코딩 없음.

| 단계 | 워크플로 | 스케줄 | 동작 |
|---|---|---|---|
| **계산** | `daily_screen.yml` | 평일 **08:30 UTC** (17:30 KST, KRX 마감 후) | 수집+채점+그라운딩 → 렌더된 알림을 `notify_queue` 저장 (발송 X) |
| **전송** | `daily_notify.yml` | **+13h = 21:30 UTC** (다음날 06:30 KST, 비서머타임 미국 마감+30분) | `notify_queue`의 **최신 미발송** 페이로드를 그대로 발송 |

- **같은 KRX 거래일**을 두 런이 공유 → 날짜 불일치 제거. 무거운 LLM은 저녁에 끝나 아침 전송은 지연 0.
- **채점 지연 폴백**: 계산이 전송창(21:30 UTC)을 넘겨 끝나면(`_missed_notify_window`) 계산 런이 **즉시 자체 발송**(중복은 dedup).
- **전송 시각 게이트**: 실제 발송 여부는 `market_clock.py`가 **미국 동부 실시각**(`America/New_York`, DST 자동)으로
  '정규장 마감(16:00 ET)+30분' 창에서만 통과시켜, cron이 엉뚱한 시각에 발화해도 잘못된 시각 전송을 막는다.
  (수동 실행 `workflow_dispatch`·`FORCE_RUN=1`·`--end-date`·`--compute`·`--submit-batch`는 게이트 우회.)
- **영속화**: SQLite를 `actions/cache`(key `dhandho-db-${run_id}`, restore-keys `dhandho-db-`)로 런 간 공유.

### 기타 워크플로
- `ci.yml` — 푸시/PR 시 테스트.
- `dump_grounding.yml`(수동) — 저장된 그라운딩(최신 거래일)을 로그로 출력(점검용, 발송 없음).
- `llm_check.yml` — LLM 라이브 점검. `llm_batch.yml` — 선택적 배치 제출(수동).
- `connectivity_check.yml` — KRX/DART/Telegram 연결 점검.

## 9. 저장소 스키마 (`storage/db.py`, SQLite)

| 테이블 | 용도 |
|---|---|
| `prices` | 전 종목 일봉 OHLCV + `rsi14` (PK: ticker,date) |
| `universe` | 종목 메타 + kind(normal/financial/holding/reit) + excluded |
| `collection_log` | 수집 멱등/catch-up (PK: date,market) |
| `scores` | A~F·총점·gate_pass·signal·flags (PK: ticker,date) |
| `signals_log` | 알림 중복방지 (BUY / `__DIGEST__` 센티넬) |
| `policy_timeline` | 상법 타임라인(E2/E3 촉매) |
| `qual_grounding` | LLM 그라운딩: item(B4/D2/D3/F1/F3 + `_DROP`/`_SELECT`/`_EVIDENCE`) score·grounded·reason·model |
| `llm_batch` | Batch API 상태(date PK, batch_id, applied) |
| `notify_queue` | 2단계 분리: 렌더된 페이로드 저장 (date PK, payload, sent_at) |

- 모든 적재는 upsert(`INSERT .. ON CONFLICT`). `notify_queue`는 재계산 시 `sent_at` 초기화(미발송 복귀).

## 10. 설정 (환경변수 / GitHub Secrets·Vars)

**Vars**: `CLAUDE_MODEL`(스코어링), `CLAUDE_EXTRACT_MODEL`(추출)

| 키 | 기본값 | 용도 |
|---|---|---|

| `LIQ_COMMON_ONLY` / `LIQ_MIN_VALUE` / `LIQ_WINDOW` | 1 / 1억 / 20 | 유동성 필터 |
| `LLM_GROUNDING` | 1 | LLM 그라운딩 on/off |
